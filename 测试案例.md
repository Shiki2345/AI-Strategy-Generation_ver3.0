# 访问策略配置助手 — 测试案例 v2.1

> 配合系统提示词 v2.1 使用，用于验证部署后的模型行为是否符合预期

---

## 测试案例 1：话题边界 — 无关话题拒绝

**用户输入：**
```
帮我写一首关于春天的诗
```

**期望行为：**
- 直接回复固定拒绝语，不展开，不写诗
- 预期输出：`我是访问策略配置助手，只能帮你配置和管理访问策略。如果你有策略需求，请描述你想限制的页面、人员和时段。`

**验证点：**
- [ ] 是否一句话打回
- [ ] 是否没有生成任何诗歌内容
- [ ] token 消耗是否极低

---

## 测试案例 2：话题边界 — 相关但不是策略

**用户输入：**
```
亚马逊广告怎么投？CPC一般设多少合适？
```

**期望行为：**
- 拒绝回答运营问题，引导回到策略配置
- 预期输出：`我是访问策略配置助手，只能帮你配置和管理访问策略。如果你有策略需求，请描述你想限制的页面、人员和时段。`

**验证点：**
- [ ] 是否拒绝回答广告投放问题
- [ ] 是否没有给出任何CPC建议

## 测试案例 3：话题边界 — 看似无关实为背景信息

**用户输入：**
```
我们公司有7个组，每组管不同的店铺，其中6组比较特殊，他们的case不想让别人看到
```

**期望行为：**
- 不拒绝，识别为策略背景信息 + 数据隔离需求
- 进入 conditionally_supported 流程，先问"6组是否使用独立的平台账号？"

**验证点：**
- [ ] 是否没有被误杀为无关话题
- [ ] 是否识别出数据隔离需求
- [ ] 是否主动追问账号结构

---

## 测试案例 4：正常流程 — 简单屏蔽

**用户输入：**
```
禁止客服访问广告页面
```

**期望行为：**
- 识别为 URL 级屏蔽，平台支持
- 进入多轮收集，逐项追问：生效时段？生效平台账号？审批限制？
- 不假设默认值，用户没说的就问

**验证点：**
- [ ] 是否主动追问缺失参数（时段、账号、审批）
- [ ] 是否没有直接假设"一直生效"而是询问用户
- [ ] 最终输出的 URL 是否用了通配符 https://*.amazon.*/advertising/*
- [ ] 生效成员是否写了具体操作路径（角色tab → 客服人员）

---

## 测试案例 5：正常流程 — 可访问模式

**用户输入：**
```
运营可以看账户详情页，但不能复制里面的内容，也不能下载
```

**期望行为：**
- 识别为 URL「可访问」模式 + 操作行为限制
- 追问缺失参数
- 输出中 url_rule 的 mode = "可访问"，附带 behavior_controls

**验证点：**
- [ ] mode 是否为"可访问"而非"已限制"
- [ ] behavior_controls 中复制是否为"限制且记录"
- [ ] behavior_controls 中文件下载是否为"限制且记录"
- [ ] 其他未提及的操作（打印、鼠标点击等）是否追问或使用合理值

---

## 测试案例 6：条件支持 — 数据隔离

**用户输入：**
```
我想让6组的成员开的case别人不能看
```

**期望行为：**
- 命中 conditionally_supported（数据隔离）
- 不直接判为不支持，先问"6组是否使用独立的平台账号？"
- 如果用户回答"有独立账号"→ 继续收集参数，用反选模式
- 输出中生效成员应为"指定成员 → 部门tab → 选中除「6组」外的所有部门"

**验证点：**
- [ ] 是否先追问账号结构
- [ ] 是否没有直接拒绝说"平台做不到"
- [ ] 如果有独立账号，最终输出的生效成员是否用了反选模式
- [ ] url_rule 是否为 https://*.amazon.*/cu/case-lobby*

---

## 测试案例 7：平台不支持

**用户输入：**
```
我们所有组共用一个卖家账号，我想让每个人只能看到自己创建的订单，别人的订单看不到
```

**期望行为：**
- 识别为数据行级过滤 + 共用账号 → 平台不支持
- 告知用户此需求超出平台能力，给出替代方案
- 不生成任何表单输出

**验证点：**
- [ ] 是否正确识别为"共用账号 + 行级过滤"
- [ ] 是否告知了替代方案（业务系统后端实现）
- [ ] 是否没有生成 JSON 输出

---

## 测试案例 7b：平台不支持 — 隐藏页面上某个字段

**用户输入：**
```
我想让运营在订单页面上看不到客户的收货地址和电话号码，其他信息正常显示
```

**期望行为：**
- 识别为字段级权限控制 → 平台不支持
- 用大白话解释：这属于业务系统内部的权限管理，不是浏览器能控制的
- 引导用户找店铺管理员，在业务系统的「员工权限」模块中设置
- 不生成表单输出

**验证点：**
- [ ] 是否识别为字段级控制（不是页面级）
- [ ] 是否没有用"Row-Level Security"等技术术语
- [ ] 是否告知了找谁（店铺管理员）、去哪里操作（业务系统员工权限模块）
- [ ] 是否没有生成 JSON 输出

---

## 测试案例 7c：平台不支持 — 多级审批链

**用户输入：**
```
我希望员工申请访问财务页面时，先由主管审批，主管通过后再由经理审批，经理通过后BOSS最终确认，三级都通过才能访问
```

**期望行为：**
- 识别为多级审批链 → 平台不支持（平台仅支持"仅boss可审批"和"不允许申请"两个选项）
- 用大白话解释平台的审批能力边界
- 引导用户找IT管理员在OA系统（钉钉/飞书/企业微信）中配置多级审批
- 主动提出"我能帮你做的"：如果两种模式能满足需求，可以配置"仅BOSS可审批"

**验证点：**
- [ ] 是否正确识别为多级审批需求
- [ ] 是否解释了平台只支持两种审批模式
- [ ] 是否引导用户去OA系统配置
- [ ] 是否主动提出了平台能做的替代方案（仅BOSS审批）
- [ ] 是否没有生成 JSON 输出

- 不生成表单输出

**验证点：**

- [ ] 是否告知找技术团队处理
- [ ] 是否没有生成 JSON 输出

## 测试案例 7e：平台不支持 — 边界模糊的需求

**用户输入：**
```
我想让新来的实习生只能看到分配给他的那几个产品的数据，其他产品的数据他看不到
```

**期望行为：**
- 识别为数据行级过滤（按产品分配过滤），平台不支持
- 先确认账号结构（是否实习生有独立账号）
- 如果共用账号 → 解释平台做不了按产品过滤数据，引导找技术负责人在ERP/业务系统中配置
- 如果有独立账号但需求本质还是"同一页面按数据行过滤" → 仍然做不了，因为即使屏蔽整个库存页面，也无法做到"只显示部分产品"

**验证点：**
- [ ] 是否先追问账号结构
- [ ] 是否正确判断为数据行级过滤（不是简单的页面屏蔽）
- [ ] 是否解释清楚"屏蔽整个页面"和"只显示部分数据"是两回事
- [ ] 是否引导找技术负责人/业务系统服务商

## 测试案例 8：批量角色策略

**用户输入：**
```
我需要按角色设置权限：
- 产品上架专员：只能用Inventory模块
- 广告推广专员：只能用广告模块
- 客服：只能看消息和订单
- 财务：只能看财务报表
请帮我批量生成策略
```

**期望行为：**
- 识别为批量角色策略需求
- 针对缺失信息统一追问（如：时段、审批、客服订单是只读还是完全可用）
- 一次性生成4条策略的表单输出

**验证点：**
- [ ] 是否追问了缺失的公共参数（时段、审批等）
- [ ] 是否追问了客服订单的"查看权限"如何实现（可访问只读 vs 已限制）
- [ ] 是否生成了4条独立的策略 JSON
- [ ] 每条策略的 url_rule 是否都有 https:// 和通配符
- [ ] 产品上架专员的策略是否屏蔽了广告、消息、订单、财务等所有非Inventory模块

## 验证清单

### 话题边界验证
- [ ] 案例1：无关话题 → 一句话拒绝
- [ ] 案例2：相关非策略 → 一句话拒绝
- [ ] 案例3：背景信息 → 不误杀，正确进入流程

### 多轮对话验证
- [ ] 案例4-8：缺失参数是否主动追问
- [ ] 是否没有擅自假设默认值

### 输出格式验证
- [ ] JSON 字段名是否对齐平台表单（策略名称、生效成员、生效时段...）
- [ ] 策略名称是否 ≤ 24字符
- [ ] 策略描述是否 ≤ 400字符
- [ ] URL 是否以 https:// 开头
- [ ] 跨站点 URL 是否用了 *.amazon.* 通配符

### 能力边界验证
- [ ] 案例6：条件支持 → 先问账号结构
- [ ] 案例7：不支持 → 拒绝并给替代方案

### Token 效率验证
- [ ] 拒绝话题的回复长度是否 < 50字
- [ ] 多轮对话的追问是否精简（不重复已知信息）
