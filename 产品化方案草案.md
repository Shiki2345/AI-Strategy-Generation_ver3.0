> 相关文档：[Demo价值定位.md](Demo价值定位.md) · [产品化架构设计.md](产品化架构设计.md) · [README.md](README.md)

## 0. 一句话结论

建议 MVP 聚焦**把 Amazon 单平台的自然语言策略生成能力嵌入紫鸟产品前端**，通过接入组织数据接口，把 demo 当前的"配置指南"升级为"**可直接应用 + 可验证 + 可预览**"的产品化工具。

**核心依据**：demo 已在真实工单用户原话上跑通"口语需求 → 可填表策略 JSON"（测试样本通过率 100%，样本详见附录 A）；工单分析显示 demo 覆盖的前 3 类失败场景占实际问题的绝大多数。

## 1. 问题定义

### 1.1 工单分析：失败并不只有一类

基于近期访问策略相关工单的样本（样本量 8 条，见附录 B），用户配置失败可归为 4 类。此前的简化归因（"不会用 + 系统 bug"）颗粒度不够，会遮蔽设计决策——因此重新分层如下：

| 失败类型 | 典型工单 | 占比感知 | AI 能做什么 | 对应 demo 能力 |
|---|---|---|---|---|
| **URL 配错**（想屏蔽 A 页但填了 B 的 URL）| ZN-386622 站点 URL 不匹配 / ZN-410465 黑白名单只生效一部分 | 最大 | 从口语翻译到正确 URL | 219 条 Amazon URL 知识库 ✅ |
| **不知道功能存在**（不知道有时段、按部门选等）| ZN-410034 限制店铺打开时段 | 中 | 主动识别意图、推荐功能 | 意图识别 ✅ |
| **平台能力边界外**（用户要的事平台做不到）| ZN-409795 合规文件链接外发 / 6 组 Case 隔离 | 中 | 主动识别并告知用户，不编配置 | v2.2 新增的能力边界过滤器 ✅ |
| **概念错误**（以为 A 的配置对 B 生效）| ZN-408653 5 个店铺 2 个没生效 | 小但危险 | ❌ 生成层面解不了 | **当前 demo 盲区** |

> **样本限制**：本分析基于 8 条工单，样本偏小。最大类别"URL 配错"是感知判断，非量化结论。扩测到 30+ 条定量验证是 P0 后续动作（见 §7）。

### 1.2 痛点本质：翻译问题，且翻译失败时没人提醒

工单里两种主要失败模式：

1. **不会配**——用户主动提工单求助
2. **配了没生效**——用户自以为配对了，其实没配对。这是冰山沉默面：不提工单，但真实存在

两种失败都指向一个本质问题：用户有一个意图（"禁止客服看广告"），但翻译不到平台的配置模型（URL × 角色 × 时段 × 操作权限），且翻译错了时**没有机制告诉他错了**。

## 2. 目标用户

### 2.1 画像锁定

目标用户不是 IT 管理员，也不是熟练运营主管，而是**业务强、IT 弱的决策者**：

- 跨境电商中小老板 / 部门 BOSS / 非技术背景业务主管
- 知道要什么结果（"客服别看广告"），不知道平台有什么功能、该填什么字段
- 配一次策略是低频行为（一季度几次），没有动力学完整文档
- 触发动机往往是"出事之后"：被薅羊毛、数据外泄、老板要求审计、招新员工

### 2.2 使用场景

| 维度 | 描述 |
|---|---|
| 触发点 | 新招员工 / 组织调整 / 安全事件 / 合规要求 |
| 频率 | 低频（非日常操作）|
| 当前替代方案 | 看文档自行配置 / 提工单找紫鸟客服定制 |
| 期望体验 | 说清楚意图就能拿到可用配置，不用学平台字段 |

> **待验证**：上述画像来自工单样本推理，未经定性访谈确认。下一步建议访谈 1-2 名一线客服，核对"提工单定制策略的客户"画像是否与上述一致。

## 3. 价值主张

**核心主张**：**降低配置门槛**（让不懂的人也能配对），而非提升熟手效率。

| 序 | 价值 | 适用场景 |
|---|---|---|
| 1 | **降门槛** | 不会配的用户 |
| 2 | **防错** | 配了没生效的用户 |
| 3 | **省时间** | 熟手配置 |

这个优先级决定所有后续取舍：
- 入口不能藏在"高级设置"深处（小白用户不会点）
- UI 不能要求用户先选"策略类型"（小白用户不知道有哪几种）
- 生成后必须用大白话说明实际效果（不用技术术语）

## 4. Demo 验证结果

### 4.1 核心命题已验证

Demo（v2.3）在真实工单的用户原话样本上跑通了"口语 → 策略 JSON"转换，输出字段与平台表单 1:1 对齐。

**证据**：样本通过率 100%（通过定义：AI 输出能直接填入平台表单，无须人工改字段）。详细案例对照表见附录 A。

> **这一节是方案最硬的论据**——不是"假设 AI 能做 X"，是"已经在真实工单上做到了 X"。

### 4.2 能力边界（已量化）

| 场景 | 覆盖率 | 说明 |
|---|---|---|
| 角色模板化配置 | ~85% | 6 种预制角色直接输出 |
| 单页屏蔽 / 审批 | ~70% | URL 在 219 条知识库内时可用 |
| 时间条件策略 | ~90% | 预设完善 |
| 不可实现需求识别 | ~80% | v2.2 能力边界过滤器有效 |
| 数据隔离 | ~50% | 依赖用户提供账号结构 |
| 复合需求 | ~40% | 需拆多条策略 |
| 非标页面 | ~30% | 超出 219 条知识库时无法准确生成 |

### 4.3 Demo 无法验证的（正是下一阶段要解决的）

| 维度 | 为什么 demo 验证不了 |
|---|---|
| 真实用户 UX | VSCode + Claude Code 环境对非技术用户不友好 |
| 组织数据集成 | Skill 全靠手输 |
| 多平台泛化 | 仅做了 Amazon |
| 成本 / ROI | 用户付 Claude 订阅费，产品化成本结构完全不同 |
| 反馈闭环 | 策略生效后效果如何，Skill 感知不到 |

## 5. 产品化方案：能力升级三件事

产品化不是简单搬迁 demo，而是借助"产品内置"的位置做 demo 做不了的事。核心升级有三件。

### 5.1 升级一：组织数据自动拉取

**解决的问题**：用户手输部门/成员/账号不准、经常写不全。

Demo 阶段 AI 只能靠用户描述"我们公司有哪些部门/成员"——用户描述不全或错，AI 就输出错。产品化后：

- 前端直接调紫鸟平台接口获取组织架构（部门、成员、角色、平台账号）
- AI 拿到真实上下文，不再依赖用户手输
- 组织数据前端拉一次，双向复用：既渲染选择器 UI，也塞进 AI 请求

**对应工单**：部分解决 ZN-408653 类"账号结构错乱"问题。

**前置调研项**：紫鸟平台接口具体能返回哪些字段？（见 §6.2）

### 5.2 升级二：生效范围可视化

**解决的问题**：用户无法验证 AI 的判断是否符合自己的意图。

Demo 输出的是"生效成员：指定成员 → 角色 tab → 选中：客服人员"——用户得自己去平台数是哪几个人、对不对。产品化后：

- 界面直接显示"已选中 3 名客服：张三、李四、王五（可取消选择）"
- AI 的判断从"说明书"变成"可见选择"，用户在提交前就能核对

**对应工单**：解决"以为配对了其实没对"类沉默失败的第一道关卡。

**设计原则**：AI 选完用户可以一键修改/取消，AI 不是黑盒。

### 5.3 升级三：提交前影响预览

**解决的问题**：概念错误类失败——这是当前 demo 完全无法覆盖的盲区。

Demo 生成完 JSON 后就结束了，不知道这条策略落到用户真实环境会发生什么。产品化后，生成后不直接提交，先做一次"假设分析"：

- 本条策略会影响 **X 个账号、Y 名成员**
- 会拦截访问 **以下页面列表**
- 与现有 **N 条策略** 是否存在冲突

**对应工单**：直接解决 ZN-408653 "5 店铺 2 个没生效" 类根因——用户在提交前看到"当前策略只影响 3 个账号"，立刻能意识到自己理解有误。

**前置调研项**：平台接口能否返回"已有策略列表"、"账号归属关系"？深度取决于接口能力。

### 5.4 入口与现有产品的关系

**不替代手动新增，作为额外选项并列**。

在「策略管理」模块新增 "AI 辅助生成" 入口，与"手动新增"并排；生成结果与手动新增落到同一张表单，用户可任意编辑后保存。

| 选择原因 |
|---|
| 降低学习成本：熟手可保持原有路径 |
| 保留退路：AI 失败时用户能切回手动 |
| 便于灰度：老用户不受影响，新功能按需开启 |

## 6. MVP 范围

### 6.1 做什么（v1）

| 能力 | 优先级 | 备注 |
|---|---|---|
| Amazon 自然语言 → 策略 JSON | P0 | Demo 已验证，直接迁移 |
| 组织数据接口接入 | P0 | 升级一 |
| 生效范围可视化 | P0 | 升级二 |
| 能力边界识别（拒绝不支持的需求）| P0 | v2.2 已有 |
| 提交前影响预览 | P1 | 升级三，取决于平台接口支持度 |
| 角色模板快捷入口 | P1 | 6 种预制角色一键使用 |

### 6.2 不做什么（明确砍掉）

| 不做 | 理由 |
|---|---|
| TikTok / Walmart 多平台 | Amazon 的 219 条 URL 知识库是差异化壁垒，盲目扩平台会摊薄质量；留给 v2 |
| 自动填表 / 一键部署 | 风险大回报小，停在"生成 → 用户确认 → 保存"足够 |
| 策略生效后的监控审计 | 超出产品范围 |
| 白名单、元素级限制 | 平台本身不支持，demo 已主动识别 |
| 复合需求一次生成多条 | demo 已知 ~40% 覆盖率，v1 引导用户分条配置 |

### 6.3 成功指标（建议）

| 指标 | 衡量方式 | 目标 |
|---|---|---|
| AI 生成成功率 | 用户点击"AI 生成"后，最终保存策略的比例 | ≥ 70% |
| 编辑率 | 用户对 AI 输出手动修改的字段占比 | < 30%（越低代表 AI 越准）|
| 工单减量 | 策略配置类工单月度数量 | 相比基线下降 ≥ 30% |
| 静默失败识别率 | 预览功能拦截的"配错作用范围"次数 | 可观测即达标 |

> 上述指标是建议值，具体阈值待与 mentor 对齐。

## 7. 开放问题（主动列出）

### 7.1 待验证假设

| 假设 | 当前证据强度 | 如何验证 |
|---|---|---|
| "URL 配错"是最大失败类别 | 弱（8 条工单目测）| 扩测 30+ 条工单，定量分类 |
| 目标用户是"业务强 IT 弱的决策者" | 弱（工单推理）| 访谈 1-2 名一线客服 |
| 价值优先级：降门槛 > 防错 > 省时间 | 中（与客服电话沟通确认方向，未量化）| MVP 灰度后用户访谈 |

### 7.2 待调研技术项

- **平台接口能力**：紫鸟平台能返回哪些组织数据？账号结构（独立/共用）、已有策略、访问日志能否拉取？这直接决定"生效预览"能做到什么深度
- **前端 / 后端技术栈**：见 [产品化架构设计.md](产品化架构设计.md) §7 P0 调研项
- **AI 调用方案**：Claude API 账号、额度、审批流程

### 7.3 风险点

**羊毛党 / 滥用**：AI 降低配置门槛，是否会被恶意利用？
- 当前判定：MVP 用户是 BOSS / IT 管理员，不是被管理对象，风险低
- 缓释：生成记录可审计、异常频率可告警（技术侧实现，PM 只提要求）
- 推进到 v1 上线前再评估

**概念错误盲区的剩余风险**：即使做了预览，仍有部分用户看不懂预览
- 这是用户教育问题，不全是产品能解
- 缓释：预览用大白话描述"会发生什么"，不用技术术语

**LLM 幻觉**：AI 输出的字段或 URL 可能是编造的
- 缓释：后端对 JSON 做合法性校验（Schema + URL 白名单）再返回前端
- Demo 里的 `validate_policy.py` 可直接迁移
