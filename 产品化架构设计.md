# 产品化架构设计

> 版本：v0.1（草案）· 日期：2026-04-22
> 文档性质：下一阶段迁移方案的技术讨论纪要，待确认事项多，未进入开发阶段

---

## 1. 背景

当前 v2.3 交付形态是 **Claude Code Skill**——用户在 VSCode + Claude Code 插件里对话触发。这种形态适合做 demo 和小范围试用，但有两个本质局限：

1. **用户必须在 Claude 环境里**，无法嵌入紫鸟现有产品，普通用户触达门槛高
2. **AI 不知道用户实际环境**（部门、成员、角色需用户自行描述），对话容易卡住

下一阶段目标：**把策略生成能力做成紫鸟产品内置功能**，由前端页面承载，AI 能直接读取用户在平台已配置的组织架构、角色、平台选择等信息。

同时产品能力要从 **Amazon 单平台** 扩展到 **Amazon + TikTok + Walmart 三平台**。

---

## 2. 目标形态

### 业务目标

让用户在紫鸟产品内的「策略管理」模块中：

1. 点击「AI 生成策略」入口
2. 用自然语言描述需求（例：「不让财务进广告后台」）
3. AI 结合用户的组织架构 + 所选平台，生成可直接保存的策略 JSON
4. 用户确认/微调后一键保存

### 技术目标

- 前端嵌入现有产品，复用现有 UI 体系和登录态
- 组织架构数据实时从平台接口读取（不再靠用户手输）
- 支持多平台模板切换，每个平台一套知识库
- AI 调用走自建后端，保证 API Key 安全和成本可控

---

## 3. 现状与差距

| 维度 | 当前（Skill 形态） | 目标（产品内置） |
|------|-------------------|-----------------|
| 运行环境 | VSCode + Claude Code 插件 | 紫鸟产品网页前端 |
| 交互形式 | 对话式 | 表单 + 对话混合 |
| 组织数据来源 | 用户手输 | 调用平台接口实时读取 |
| 平台支持 | Amazon | Amazon + TikTok + Walmart |
| AI 运行位置 | Anthropic（Claude 自家） | 自建后端 → Claude API |
| Prompt/知识库管理 | SKILL.md + references/ | 后端 prompt 模板 + 知识库文件 |
| 鉴权 | 无（靠 Claude 账号） | 复用产品现有登录态 |
| 成本/审计 | 用户付 Claude 订阅费 | 产品方付 API 费，可审计每次调用 |

## 5. 关键设计决策

### 决策 1：AI 调用必须走自建后端

**不能**让前端直接调 Claude API，原因：

1. **密钥安全**：前端代码用户 F12 可见，API Key 泄露即耗尽额度
2. **成本控制**：后端可做限流、配额、按公司/用户计费
3. **Prompt 保护**：精心调优的 prompt 和知识库是核心资产
4. **审计日志**：合规要求能追溯「谁、何时、基于什么输入、生成了什么策略」

### 决策 2：组织数据由前端直接调自家接口

**不走后端中转**，原因：

- 自家平台接口的鉴权跟随用户登录态（cookie/token），前端天然有权限
- 数据是「当前用户视角下的组织架构」，后端代调反而要重建上下文
- 简单、少一跳、少一份维护代码

**例外**：如果后续需要跨接口聚合（例如 role + access_rule + member 拼在一起），再加后端聚合层。当前阶段不做。

### 决策 3：组织数据前端拉一次，双向复用

```
前端调接口一次
  ├─ 存入 store → 渲染选择器、下拉框（展示）
  └─ 调 AI 时 → 把相关子集塞进请求（AI 消费）
```

**不要分别为「展示」和「AI」各调一次**，浪费请求也容易不一致。

### 决策 4：AI 返回采用流式

策略 JSON 通常几十行，非流式会让用户干等数秒。流式方案：

- 后端：SSE（Server-Sent Events）转发 Claude API 的流式输出
- 前端：边接收边渲染，用户立刻看到 AI 开始工作

### 决策 5：Skill 不直接搬，拆成「知识库 + prompt 模板」

**Claude Skill 是 Claude 专属运行时形态，无法嵌入自建产品**。但现有 Skill 里的内容资产完全可以复用，按以下方式迁移：

| 原 Skill 产物 | 迁移后 |
|--------------|--------|
| `SKILL.md` | 后端 `prompts/system.md`，拼入 Claude API 的 system |
| `references/*.md`（平台知识） | 后端 `knowledge/platforms/<平台>/*.md`，按需拼入 prompt |
| `assets/rag_simple_db.json` | 后端 `knowledge/platforms/<平台>/rag_db.json` |
| `assets/presets.json` | 后端前端共享的预设数据 |
| `scripts/validate_policy.py` | 后端生成后的 JSON 合法性校验 |

### 决策 6：平台知识库按平台隔离

每个平台一个目录，共用的 schema/约束放 `platforms/common/`。理由：

- 不同平台表单字段、URL 规则、业务逻辑差异大
- 扩展新平台（例如后续加 Shopee）时，按目录复制+改写即可
- Prompt 拼接时按用户选择的平台加载对应知识，减少 token 浪费

## 6. 现有 Skill 资产的复用清单

原 `.claude/skills/access-policy-builder/` 目录下的资产，在产品化后的去向：

| 现有资产 | v2.3 数值 | 迁移去向 | 备注 |
|---------|----------|---------|------|
| SKILL.md 5 阶段对话流程 | 5 Phase | 后端 system prompt | 需要重写为产品形态下的指令（不再完全对话式，可能是"需求 → 直接出 JSON"）|
| Amazon URL 知识库 | 219 条 | `platforms/amazon/rag_db.json` | 原样迁移 |
| 平台能力地图 | 24 模块 + 17 指路规则 | `platforms/amazon/platform_modules.md` | 原样迁移 |
| URL 模块映射 | - | `platforms/amazon/url_modules.md` | 原样迁移 |
| 表单 schema | - | `platforms/amazon/form_schema.md` | 原样迁移 |
| 角色模板 | 6 种 | `platforms/amazon/role_presets.json` | 原样迁移，TK/Walmart 需各自建 |
| 预设 | 5 类 | `common/presets.json` | 大部分通用（时段、浏览器控制），部分平台相关 |
| 校验脚本 | `validate_policy.py` | 后端校验模块 | 原样迁移，按平台分支 |

**TK 和 Walmart 需要从零建立**：URL 知识库、平台能力地图、表单 schema、角色模板——这是整个扩展工作量的大头。

## 8. 下一步行动

按依赖顺序，建议的推进节奏：

### 阶段一：调研对齐（1-2 周）

目的：填完第 7 节所有 P0 项，形成确定的技术选型。

产出：
- 技术栈清单（前端框架、后端语言、部署方式、AI 接入方式）
- 组织架构接口清单（路径、入参、返回结构）
- 与现有产品的集成点确认（入口位置、鉴权对接、样式规范）

### 阶段二：统一设计（1 周）

目的：完成第 7 节 P1 设计项，把跨平台架构定下来。

产出：
- 策略 JSON Schema（含三平台通用部分 + 各自扩展）
- `/api/ai/generate-policy` 接口文档
- 前端页面原型（Figma 或手绘皆可）

### 阶段三：单平台 MVP（2-3 周）

目的：Amazon 通路打通，端到端能跑一个完整流程。

关键里程碑：
1. 前端页面原型搭好，能渲染组织架构选择器
2. 后端 `/api/ai/generate-policy` 接口跑通，能调 Claude API
3. Amazon 知识库从 Skill 迁移到后端
4. 端到端生成一条真实策略并保存

### 阶段四：平台扩展（2-4 周，视内容准备速度）

- TK 平台知识库建设 + 接入
- Walmart 平台知识库建设 + 接入
- 多平台切换联调

### 阶段五：试用与迭代（持续）

- 灰度给内部用户
- 收集失败 case 反哺 prompt 和知识库
- 关注复合需求、非标页面等当前 v2.3 已知的风险场景

## 9. 开放问题（尚未讨论）

这些问题本轮讨论暂未覆盖，推进到相应阶段时需要补齐：

- **多语言支持**：TK/Walmart 面向的用户是否有英文需求？prompt 和 UI 是否要双语？
- **生成结果的持久化**：生成的 JSON 除了当前策略，是否要沉淀为模板供后续复用？
- **失败 case 反馈循环**：用户微调生成结果后，如何回流到知识库/prompt？
- **多账号/多店铺场景**：一个用户管多个店铺时，组织架构是按店铺隔离还是合并？
- **权限边界**：谁能用 AI 生成功能？是否按角色开放？
- **降级策略**：Claude API 故障或超限时的降级方案（缓存？规则引擎兜底？）
