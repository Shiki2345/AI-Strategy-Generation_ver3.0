# Demo 价值定位

> 版本：v1.0 · 日期：2026-04-22
> 文档性质：对当前 v2.3 Skill 形态 demo 的价值回顾与定位说明

---

## 核心结论

当前 demo 本质上是**多层次的可行性验证**，不止「AI 能不能生成策略」这一件事。

一句话定位：**用最低成本，把「AI 生成访问策略」这条路径的核心假设跑通，避免直接做产品化时踩大坑。**

---

## Demo 验证了什么

| 验证维度 | Demo 给出的答案 |
|---------|---------------|
| **核心命题**：LLM 能否把自然语言需求转成结构化策略 JSON | ✅ 可以，且质量够用（角色模板 85%、时间条件 90%） |
| **工程可行**：用 prompt + 知识库的方式编码领域经验，是否比直接问 LLM 更可靠 | ✅ 可以，Skill 方案让 LLM 不再靠"即兴发挥" |
| **能力边界**：哪些需求可以做好、哪些会翻车 | ✅ 搞清楚了——复合需求、非标页面、数据隔离是坑 |
| **内容需求量**：要准确生成需要多少专家知识 | ✅ 量出来了——219 条 URL、24 模块、17 条指路规则、6 套角色模板 |
| **领域适配度**：策略配置这个场景适不适合 AI 辅助 | ✅ 适合——把门槛从"理解每个表单字段"降到"说清楚想限制谁做什么" |

---

## Demo 没法验证的（所以才需要下一阶段）

| 未验证维度 | 为什么 Demo 验证不了 |
|----------|-------------------|
| **真实用户 UX** | Claude Code 环境对真实用户不友好，用户不会为此学 VSCode |
| **组织数据集成** | Skill 里全靠手输，产品场景必须对接真实平台数据 |
| **多平台泛化** | 只做了 Amazon，TK/Walmart 的领域知识和差异点还没摸清 |
| **成本与 ROI** | 用户付 Claude 订阅费，产品化后产品方自己付 API 费，成本结构完全不同 |
| **规模化运营** | 1 个 demo 用户 vs 1000 个真实用户，prompt 稳定性、边界 case 完全是另一个量级 |
| **反馈闭环** | 用户改完 JSON 填到平台后效果如何，Skill 根本感知不到 |

---

## Demo 实际做了三件事

1. **可行性验证**（核心命题能不能成）— 这是主任务
2. **领域知识盘点**（需要多少内容、什么结构）— 副产品，但对下一阶段非常值钱
3. **风险识别**（哪些场景会翻车）— 为下一阶段的产品边界划线

从投入产出看，这是个**很划算的前置动作**。Demo 跑通之后，下一阶段就不再是「赌 AI 能不能做」，而是「怎么把已验证的能力工程化」——**风险降了一个维度**。

---

## 对下一阶段的意义

推进产品化时，Demo 成果**不用重新验证**，可以直接作为前置结论使用：

- **核心命题可行** → 不用再争「要不要做」
- **知识库结构有效** → 直接迁移，不用重新设计
- **风险 case 已知** → 产品设计时提前规避（比如复合需求在 UI 上就引导用户拆多条）

---

## 结论

Demo 的使命基本完成了。继续在 Skill 形态上打磨，边际收益递减，价值已经不如推进下一阶段（产品化）大。
